# Security Policy

## Supported Versions

| Version | Supported          |
| ------- | ------------------ |
| 1.0.x   | :white_check_mark: |

## Security Features

This bot implements several security measures:

### Authentication & Authorization
- User ID whitelist for authorized users only
- All Telegram commands require authentication
- Unauthorized access attempts are logged

### Environment Security
- Sensitive credentials stored in `.env` file
- `.env` file excluded from git via `.gitignore`
- No hardcoded secrets in source code

### Rate Limiting
- API calls are rate-limited to prevent abuse
- Delays between token analyses
- Cache implementation for macro data

### Error Handling
- Graceful degradation on API failures
- Comprehensive error logging
- No sensitive data in error messages

## Best Practices

### Setting Up Securely

1. **Never commit `.env` file**
```bash
# Always in .gitignore
.env
```

2. **Use strong bot tokens**
- Generated by Telegram's @BotFather
- Never share publicly
- Regenerate if compromised

3. **Restrict user access**
```env
ALLOWED_USER_IDS=123456789  # Only you
```

4. **Run in isolated environment**
- Use Docker container or dedicated VPS
- Separate from other services
- Regular security updates

5. **Monitor logs**
```bash
# Check for unauthorized access attempts
grep "Unauthorized" logs/*.log
```

### API Security

1. **Public APIs**
- Uses public endpoints (no API keys stored)
- Implements request timeouts
- Handles rate limiting gracefully

2. **Data Privacy**
- No user data stored
- No trading data logged
- Minimal data retention

### Code Security

1. **Dependencies**
```bash
# Regularly update dependencies
npm audit
npm update
```

2. **Input Validation**
- All user inputs are validated
- Environment variables are checked
- Type safety with TypeScript

## Reporting a Vulnerability

If you discover a security vulnerability:

1. **DO NOT** open a public issue
2. Email details to [your-email@example.com]
3. Include:
   - Description of the vulnerability
   - Steps to reproduce
   - Potential impact
   - Suggested fix (if any)

We'll respond within 48 hours and work on a fix.

## Security Checklist

Before deploying:

- [ ] `.env` file created with unique credentials
- [ ] `ALLOWED_USER_IDS` configured
- [ ] Bot token is fresh and secure
- [ ] Running in isolated environment
- [ ] Logs monitored for suspicious activity
- [ ] Dependencies up to date (`npm audit`)
- [ ] `.env` not committed to git
- [ ] Server has firewall configured
- [ ] Regular backups configured (if storing data)

## Disclosure Policy

- Report received: 0-2 days acknowledgment
- Fix developed: 2-7 days
- Security patch released: 7-14 days
- Public disclosure: 14-30 days after patch
